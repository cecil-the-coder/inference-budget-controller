## 1. Module Structure

```
fox-engine/
├── Cargo.toml
├── build.rs                     # llama.cpp cmake + bindgen (transplanted verbatim)
└── src/
    ├── lib.rs                   # pub re-exports; crate root
    ├── engine/
    │   ├── mod.rs               # InferenceEngine struct + run_loop
    │   ├── model.rs             # Model trait, LlamaCppModel, ModelConfig, Logits,
    │   │                        #   InferenceRequestForModel
    │   └── ffi.rs               # Raw bindgen types (pub(crate) only)
    ├── scheduler/
    │   ├── mod.rs               # Scheduler struct, prefix cache, scheduling logic
    │   └── batch.rs             # InferenceRequest, SamplingParams, Token, StopReason,
    │                            #   RequestState, ScheduledBatch, PageTable
    ├── kv_cache/
    │   └── mod.rs               # KVCacheManager, PageTable, BlockId,
    │                            #   compute_block_hash, prompt_block_hashes
    ├── registry/
    │   └── mod.rs               # ModelRegistry, EngineEntry, RegistryConfig, LoadParams
    └── metrics/
        └── mod.rs               # EngineMetrics (renamed from Metrics to avoid collision
                                 #   with controller-side metrics)
```

### What each module contains

| Module | Responsibility |
|---|---|
| `engine` | `InferenceEngine`: owns `Arc<dyn Model>`, `Arc<Scheduler>`, `Arc<KVCacheManager>`; drives the continuous-batch run loop; exposes `submit_request`, `tokenize`, `embed`, `apply_chat_template` |
| `engine::model` | `Model` trait (sync FFI calls); `LlamaCppModel` (the llama.cpp impl); `ModelConfig`; `Logits`; `InferenceRequestForModel` |
| `engine::ffi` | Raw C bindings generated by bindgen — `pub(crate)` only, never part of the public API |
| `scheduler` | `Scheduler`: waiting queue, running batch, LIFO preemption, LRU prefix cache, `schedule_step` |
| `scheduler::batch` | All data types that cross the engine↔scheduler boundary: `InferenceRequest`, `SamplingParams`, `Token`, `StopReason`, `RequestState`, `ScheduledBatch` |
| `kv_cache` | `KVCacheManager`: free-list block allocator with ref-counted CoW; `PageTable`; block hashing |
| `registry` | `ModelRegistry`: LRU model cache, `get_or_load`, `unload`, `list_loaded`, `is_loaded`; `EngineEntry` |
| `metrics` | `EngineMetrics`: all Prometheus counters/gauges/histograms for one engine instance |

## 2. ModelRegistry API

### Types

```rust
// fox-engine/src/registry/mod.rs

use std::collections::HashMap;
use std::path::PathBuf;
use std::sync::Arc;
use anyhow::Result;

/// Parameters that can differ between load calls for the same model name.
/// The controller passes `gpu_memory_bytes` from the GTT budget here.
#[derive(Debug, Clone)]
pub struct LoadParams {
    /// GPU memory budget in bytes, from the controller's GTT allocation.
    /// Passed directly to KVCacheManager::new as `gpu_memory_bytes`.
    pub gpu_memory_bytes: usize,
    /// Fraction of gpu_memory_bytes to use for KV blocks (default: 0.85).
    pub gpu_memory_fraction: f32,
    /// Maximum number of concurrent requests this engine will service.
    pub max_batch_size: usize,
    /// Hard context-window limit in tokens (passed to llama.cpp on load).
    pub max_context_len: u32,
    /// KV cache block size in tokens (default: 16).
    pub block_size: usize,
}

impl Default for LoadParams {
    fn default() -> Self {
        Self {
            gpu_memory_bytes: 8 * 1024 * 1024 * 1024, // 8 GiB
            gpu_memory_fraction: 0.85,
            max_batch_size: 16,
            max_context_len: 4096,
            block_size: 16,
        }
    }
}

/// Static registry-level configuration that does not change between loads.
#[derive(Debug, Clone)]
pub struct RegistryConfig {
    /// Directory scanned for GGUF model files.
    pub models_dir: PathBuf,
    /// Maximum number of models held in memory simultaneously (LRU eviction).
    pub max_models: usize,
    /// Optional Prometheus metrics shared across all loaded engines.
    pub metrics: Option<Arc<crate::metrics::EngineMetrics>>,
    /// Alias map: short name → canonical file stem (e.g. "llama3" → "Meta-Llama-3-8B-Q4_K_M").
    pub aliases: HashMap<String, String>,
}

/// A loaded model + its running engine loop.
pub struct EngineEntry {
    pub engine: Arc<crate::engine::InferenceEngine>,
    /// Aborted when dropped (LRU eviction or explicit unload).
    loop_handle: tokio::task::JoinHandle<()>,
    /// The budget that was active when this model was loaded.
    /// Stored so the controller can detect when the budget changed enough to
    /// warrant an unload + reload with a new KVCacheManager.
    pub loaded_params: LoadParams,
}

impl Drop for EngineEntry {
    fn drop(&mut self) {
        self.loop_handle.abort();
    }
}

pub struct ModelRegistry {
    engines: dashmap::DashMap<String, Arc<EngineEntry>>,
    lru: std::sync::Mutex<lru::LruCache<String, ()>>,
    config: RegistryConfig,
}
```

### Public API methods

```rust
impl ModelRegistry {
    pub fn new(config: RegistryConfig) -> Self;

    /// Return the cached engine for `name`, or load it from disk using `params`.
    ///
    /// - If already loaded: returns immediately (O(1) DashMap lookup + LRU promote).
    /// - If not loaded: loads synchronously in a spawn_blocking task, then starts the
    ///   engine run_loop as a detached tokio task.
    /// - If at capacity: evicts the LRU entry before loading.
    ///
    /// `name` is resolved via aliases, then exact/prefix/contains matching against
    /// GGUF stems in `models_dir` (same resolution logic as current fox code).
    pub async fn get_or_load(
        &self,
        name: &str,
        params: LoadParams,
    ) -> Result<Arc<EngineEntry>>;

    /// Forcibly unload a model. Returns true if it was loaded.
    ///
    /// Dropping the EngineEntry aborts the run_loop JoinHandle, which causes
    /// LlamaCppModel's Drop impl to call llama_free on the next GC cycle.
    /// In-flight requests will receive an error on their response channel.
    pub fn unload(&self, name: &str) -> bool;

    /// All currently-loaded (stem, entry) pairs.
    pub fn list_loaded(&self) -> Vec<(String, Arc<EngineEntry>)>;

    /// True if a model with this name is currently loaded.
    pub fn is_loaded(&self, name: &str) -> bool;
}
```

### Internal loading function (private)

```rust
async fn load_engine(
    stem: &str,
    path: &std::path::Path,
    params: &LoadParams,
    metrics: Option<Arc<crate::metrics::EngineMetrics>>,
) -> Result<EngineEntry> {
    let path = path.to_path_buf();
    let stem = stem.to_string();
    let params_clone = params.clone();

    // llama.cpp model load is blocking CPU/GPU work — must not run on tokio thread.
    let model = tokio::task::spawn_blocking(move || {
        crate::engine::model::LlamaCppModel::load(
            &path,
            params_clone.max_batch_size,
            params_clone.max_context_len,
        )
    })
    .await
    .map_err(|e| anyhow::anyhow!("spawn_blocking join: {e}"))??;

    let model_config = model.model_config();
    let model: Arc<dyn crate::engine::model::Model> = Arc::new(model);

    // Budget handoff: gpu_memory_bytes comes from LoadParams (set by controller).
    let kv_cache = Arc::new(crate::kv_cache::KVCacheManager::new(
        &model_config,
        params.gpu_memory_bytes,       // <-- controller's GTT budget
        params.gpu_memory_fraction,
        params.block_size,
    ));

    let scheduler = Arc::new(crate::scheduler::Scheduler::new(
        kv_cache.clone(),
        params.max_batch_size,
    ));

    let engine = Arc::new(crate::engine::InferenceEngine::new(
        model,
        scheduler,
        kv_cache,
        stem,
        metrics,
    ));

    let loop_handle = {
        let e = engine.clone();
        tokio::spawn(async move {
            if let Err(err) = e.run_loop().await {
                tracing::error!("engine run_loop exited with error: {err}");
            }
        })
    };

    Ok(EngineEntry {
        engine,
        loop_handle,
        loaded_params: params.clone(),
    })
}
```

## 3. gpu_memory_bytes Budget Handoff

### The flow

```
controller crate
    InferenceBudgetController::reconcile()
        → reads pod.status.allocatedResources["nvidia.com/gpu-memory"]
          (or nvidia-smi / metrics-server)
        → computes gpu_memory_bytes: u64 from GTT budget
        → calls registry.get_or_load(model_name, LoadParams {
              gpu_memory_bytes: budget_bytes as usize,
              gpu_memory_fraction: 0.85,
              ..Default::default()
          })

fox-engine crate
    ModelRegistry::get_or_load(name, params)
        → load_engine(stem, path, &params, metrics)
            → KVCacheManager::new(
                  &model_config,
                  params.gpu_memory_bytes,   // <── budget arrives here
                  params.gpu_memory_fraction,
                  params.block_size,
              )
                  → compute_total_blocks(
                        gpu_memory_bytes,
                        gpu_memory_fraction,
                        block_size,
                        num_layers,
                        num_heads_kv,
                        head_dim,
                    )
                        → total_blocks = floor(
                              gpu_memory_bytes * fraction
                              / (block_size * num_layers * num_heads_kv * head_dim * 2 * 2)
                          )
```

### Budget change handling

When the controller detects the GPU memory budget has changed (e.g. another model was loaded or pod limits were adjusted), it calls:

```rust
// controller pseudocode
let entry = registry.get_or_load(name, params).await?;
if entry.loaded_params.gpu_memory_bytes != new_budget {
    registry.unload(name);                               // aborts run_loop, drops KVCacheManager
    registry.get_or_load(name, new_params).await?;       // reloads with new KVCacheManager
}
```

The `loaded_params` field on `EngineEntry` enables this comparison without any additional state on the controller side.

The KVCacheManager is **not** mutably resizable after construction — `total_blocks` is fixed at construction time (the free-list Vec is pre-allocated). Changing the budget always requires unload + reload. This is intentional: llama.cpp doesn't support dynamic context reallocation in-flight.

## 4. InferenceEngine Run Loop

### Tokio integration strategy

The engine has two worlds:
- **Async world** (tokio): request submission, streaming token delivery via `mpsc::UnboundedSender<Token>`, waiting for work via `tokio::sync::Notify`
- **Blocking world** (llama.cpp FFI): prefill and decode call `llama_decode` which blocks the OS thread for potentially hundreds of milliseconds

The rule is: **all llama.cpp FFI calls go through `tokio::task::spawn_blocking`**. The run loop itself is an `async fn` running on a normal tokio task. It calls `spawn_blocking` for each prefill batch and each decode batch, awaiting the result before proceeding.

### Run loop internals

```rust
// fox-engine/src/engine/mod.rs

impl InferenceEngine {
    /// Drive the continuous batching loop. Never returns except on unrecoverable error.
    /// Spawned as a detached tokio task by load_engine().
    pub async fn run_loop(self: Arc<Self>) -> anyhow::Result<()> {
        let mut last_prefix_hits: u64 = 0;
        let mut last_prefix_misses: u64 = 0;

        loop {
            // --- Step 1: scheduling (sync, fast, no FFI) ---
            // schedule_step() acquires three std::sync::Mutex locks momentarily.
            // It is synchronous and completes in microseconds; no need for
            // spawn_blocking. Running it inline on the async thread is correct
            // and avoids a context switch.
            let batch = self.scheduler.schedule_step();

            // --- Step 2: metrics refresh (cheap, every loop tick) ---
            if let Some(m) = &self.metrics {
                m.kv_cache_usage_ratio.set(self.kv_cache.memory_usage() as f64);
                m.queue_depth.set(self.scheduler.queue_depth() as i64);
                m.active_requests.set(self.scheduler.active_requests() as i64);
                // propagate monotonic counter deltas
                let cur_hits   = self.scheduler.prefix_hits.load(Ordering::Relaxed);
                let cur_misses = self.scheduler.prefix_misses.load(Ordering::Relaxed);
                let dh = cur_hits.saturating_sub(last_prefix_hits);
                let dm = cur_misses.saturating_sub(last_prefix_misses);
                if dh > 0 { m.prefix_cache_hits_total.inc_by(dh); }
                if dm > 0 { m.prefix_cache_misses_total.inc_by(dm); }
                last_prefix_hits   = cur_hits;
                last_prefix_misses = cur_misses;
            }

            // --- Step 3: clear KV slots for preempted sequences ---
            // clear_sequence() acquires the llama_context mutex, so runs in
            // spawn_blocking to avoid blocking the async thread.
            if !batch.preempted_seq_ids.is_empty() {
                let model = self.model.clone();
                let ids = batch.preempted_seq_ids.clone();
                tokio::task::spawn_blocking(move || {
                    for id in ids { model.clear_sequence(id); }
                }).await?;
            }

            // --- Step 4: idle wait ---
            if batch.is_empty() {
                // Notified when Scheduler::submit() is called from any thread.
                self.scheduler.wait_for_work().await;
                continue;
            }

            // --- Step 5: prefill (blocking FFI) ---
            // Runs a full llama_decode over the prompt tokens of all newly
            // admitted requests. Each request contributes its full prompt as a
            // batch of positions. Returns one (req_id, Logits, tokens_in_kv) tuple
            // per request.
            //
            // spawn_blocking rationale: llama_decode can run for 100s of ms on large
            // prompts. Blocking the tokio thread would starve the scheduler and all
            // other async tasks. Using spawn_blocking puts the work on a dedicated
            // blocking thread pool (tokio's default: 512 threads, each parked when idle).
            if !batch.prefill.is_empty() {
                match self.run_prefill(&batch.prefill).await {
                    Ok(results) => self.handle_logits(&results, true).await?,
                    Err(e) => {
                        tracing::warn!("prefill error (KV full?): {e}");
                        for id in &batch.prefill {
                            self.scheduler.mark_finished(*id, StopReason::Length);
                        }
                    }
                }
            }

            // --- Step 6: decode (blocking FFI) ---
            // Runs one decode step (single-token generation) across all currently
            // active requests. Each request contributes exactly one token position.
            // Returns one (req_id, Logits) per request.
            //
            // Why separate from prefill: llama.cpp's llama_decode treats batch
            // entries with logits=true differently. Prefill only needs the logit for
            // the LAST token of each prompt; decode needs the logit for the one new
            // position. Batching them together requires careful position management;
            // keeping them separate is simpler and has negligible throughput cost
            // because GPU utilisation during decode is compute-bound anyway.
            if !batch.decode.is_empty() {
                match self.run_decode(&batch.decode).await {
                    Ok(results) => self.handle_logits(&results, false).await?,
                    Err(e) => {
                        tracing::warn!("decode error (KV full?): {e}");
                        for id in &batch.decode {
                            self.scheduler.mark_finished(*id, StopReason::Length);
                        }
                    }
                }
            }
        }
    }

    async fn run_prefill(&self, req_ids: &[u64]) -> anyhow::Result<Vec<(u64, Logits)>> {
        // Build InferenceRequestForModel views (no lock needed: get_running is a snapshot).
        let requests = self.scheduler.get_running(req_ids);
        let model_reqs: Vec<InferenceRequestForModel> = requests.iter().map(to_model_req).collect();
        let prefix_cleanup: Vec<i32> = model_reqs.iter().filter_map(|r| r.prefix_seq_id).collect();

        let model = self.model.clone();
        let ids   = req_ids.to_vec();
        // spawn_blocking: llama_decode is blocking FFI.
        let raw = tokio::task::spawn_blocking(move || model.prefill_sync(&ids, &model_reqs))
            .await
            .map_err(|e| anyhow::anyhow!("prefill join: {e}"))??;

        // After the blocking work returns, do post-processing back on the async thread:
        // clear prefix KV slots and record actual token counts.
        for seq_id in prefix_cleanup {
            self.model.clear_sequence(seq_id);
            self.scheduler.return_prefix_seq_id(seq_id);
        }
        let result = raw.into_iter().map(|(id, logits, count)| {
            if count > 0 { self.scheduler.set_prefilled_tokens(id, count); }
            (id, logits)
        }).collect();
        Ok(result)
    }

    async fn run_decode(&self, req_ids: &[u64]) -> anyhow::Result<Vec<(u64, Logits)>> {
        // Copy-on-write guard: any block shared with the prefix cache gets
        // privatised before llama.cpp writes a new token into it.
        for req_id in req_ids {
            let reqs = self.scheduler.get_running(&[*req_id]);
            let Some(req) = reqs.first() else { continue };
            for (logical_idx, &block_id) in req.page_table.entries.iter().enumerate() {
                if self.kv_cache.is_shared(block_id) {
                    if let Some(new_id) = self.kv_cache.copy_on_write(block_id) {
                        self.scheduler.cow_update_page_table(*req_id, logical_idx, new_id);
                    }
                }
            }
        }

        let requests  = self.scheduler.get_running(req_ids);
        let model_reqs: Vec<InferenceRequestForModel> = requests.iter().map(to_model_req_decode).collect();
        let model = self.model.clone();
        let ids   = req_ids.to_vec();
        // spawn_blocking: llama_decode is blocking FFI.
        tokio::task::spawn_blocking(move || model.decode_sync(&ids, &model_reqs))
            .await
            .map_err(|e| anyhow::anyhow!("decode join: {e}"))?
    }
}
```

### spawn_blocking contract

`tokio::task::spawn_blocking` places the closure on a dedicated OS thread from tokio's blocking pool (default max: 512 threads; each blocks until the FFI call returns). The closure captures an `Arc<dyn Model>` (cheap clone), so the async task awaiting it holds no locks.

The returned `JoinHandle<Result<T>>` is double-wrapped: the outer `Result` is the join result (panics → `Err`); the inner `Result` is the FFI call's result. The `??` idiom unwraps both.

Critical invariant: **no `std::sync::Mutex` is held across `.await`**. The `per_request_state` mutex is locked, processed, and immediately dropped before any `.await` point.

## 5. Changes vs Current fox Code

### What stays exactly the same (transplant verbatim)

| Component | Notes |
|---|---|
| `engine/mod.rs` | All output filtering logic (`apply_output_filter`, `flush_pending_output`, `find_holdback_start`, `check_stop_sequences`, `trim_text_buffer`), the `run_loop`, `run_prefill`, `run_decode`, `handle_logits` |
| `engine/model.rs` | `Model` trait, `LlamaCppModel`, `ModelConfig`, `Logits`, `InferenceRequestForModel` — zero changes |
| `engine/ffi.rs` | Raw bindgen output — transplanted as-is |
| `scheduler/mod.rs` | `Scheduler` including prefix cache — transplanted as-is |
| `scheduler/batch.rs` | All types — transplanted as-is |
| `kv_cache/mod.rs` | `KVCacheManager`, `PageTable`, block hashing — transplanted as-is |
| `build.rs` | cmake + bindgen invocation — transplanted as-is |

### What changes

**1. `model_registry.rs` → `registry/mod.rs`**

The old `RegistryConfig` had `gpu_memory_bytes` and `gpu_memory_fraction` as top-level fields used for every model load. The new design separates these into `LoadParams` which is passed per-call to `get_or_load`. This lets the controller supply different budgets for each model (or update the budget on reload) without reconstructing the registry.

Old signature:
```rust
pub async fn get_or_load(&self, name: &str) -> Result<Arc<EngineEntry>>
```

New signature:
```rust
pub async fn get_or_load(&self, name: &str, params: LoadParams) -> Result<Arc<EngineEntry>>
```

`EngineEntry` gains `loaded_params: LoadParams` so the controller can detect when a reload is needed.

`aliases` moves from a constructor argument into `RegistryConfig`.

**2. `metrics.rs` → `metrics/mod.rs`, renamed to `EngineMetrics`**

Renamed to avoid collision when the controller crate imports both `fox-engine` and its own Prometheus metrics. Metric name strings change from `ferrumox_*` to `fox_engine_*`.

The `EngineMetrics` struct and `impl EngineMetrics { pub fn new() -> Result<Self> }` stay structurally identical; only the string identifiers change.

**3. Remove all `crate::cli` / `crate::api` / `crate::bin` dependencies from registry**

`resolve_model_name` used `crate::cli::list_models`. In the library crate this is re-implemented inline with `std::fs::read_dir` — there is no CLI module. The fox binary crate will import `fox-engine` and add its own CLI on top.

**4. All `crate::` references become `fox_engine::`**

Internal cross-module references already use `crate::` paths; these all remain valid because the modules are still part of the same crate. No changes needed to inter-module use statements.

**5. No Kubernetes types anywhere in fox-engine**

The current fox codebase has no k8s types, so there is nothing to remove. The controller crate is the only place that imports `kube` / `k8s-openapi`.

### How the controller crate calls the engine

```rust
// controller/src/serving.rs (sketch — lives in controller crate, not fox-engine)

use fox_engine::{EngineEntry, LoadParams, ModelRegistry, RegistryConfig};
use fox_engine::{InferenceRequest, SamplingParams};
use tokio::sync::mpsc;

async fn handle_completion(
    registry: &ModelRegistry,
    model_name: &str,
    gpu_budget_bytes: u64,
    prompt_tokens: Vec<i32>,
    max_new_tokens: usize,
    sampling: SamplingParams,
) -> anyhow::Result<impl futures::Stream<Item = fox_engine::Token>> {
    let params = LoadParams {
        gpu_memory_bytes: gpu_budget_bytes as usize,
        ..Default::default()
    };
    let entry = registry.get_or_load(model_name, params).await?;
    let engine = &entry.engine;

    let (tx, rx) = mpsc::unbounded_channel();
    let req = InferenceRequest::new(
        engine.next_request_id(),
        prompt_tokens,
        max_new_tokens,
        sampling,
        tx,
    );
    engine.submit_request(req);

    Ok(tokio_stream::wrappers::UnboundedReceiverStream::new(rx))
}
```

### Notes on omitted dependencies

- `axum`, `tower-http`, `clap`, `reqwest`, `serde`, `uuid`: all belong to the fox binary/server crate, not the engine library. The engine has no HTTP layer and no CLI.
- `tokio-stream`, `async-stream`, `futures`: the engine delivers tokens via `mpsc::UnboundedSender<Token>`; streaming wrappers are the caller's responsibility.
- `indicatif`, `crossterm`, `dialoguer`: CLI/UI concerns, not engine concerns.
- `sha2`, `hex`: block hashing uses `ahash` (already present); SHA-2 is not needed.
- `toml`: configuration parsing lives in the controller/binary crate.

## Summary of key design decisions

**LoadParams per call, not per registry**: The controller reconciles budget per-model-per-node. Passing `LoadParams` to `get_or_load` rather than baking it into a static `RegistryConfig` means the controller can request a different `gpu_memory_bytes` for each model without tearing down the whole registry. The `EngineEntry::loaded_params` field enables cheap change detection.

**KVCacheManager is immutable after construction**: `total_blocks` is fixed from `gpu_memory_bytes` at `KVCacheManager::new`. Changing the budget requires `unload` + `get_or_load`. This is correct because llama.cpp does not support dynamic context reallocation, and the free-list `Vec` pre-allocates all block slots.

**spawn_blocking for all FFI**: `llama_decode` (prefill and decode) blocks the OS thread for up to seconds on large prompts or batches. All FFI calls must run in `spawn_blocking` to keep the tokio async thread responsive for scheduling, metrics, and request submission. The closures capture only `Arc` handles and do not hold any `Mutex` guards.

**No k8s types in fox-engine**: The crate boundary is strict. `fox-engine` knows nothing about Pods, CRDs, or node budgets. The controller crate is the only place that reads the GTT budget from Kubernetes and translates it into a `LoadParams`.

**EngineMetrics renamed from Metrics**: Avoids symbol collision when controller crate defines its own `Metrics` struct and imports both crates. Metric name strings change from `ferrumox_*` to `fox_engine_*` to reflect the new crate identity.
